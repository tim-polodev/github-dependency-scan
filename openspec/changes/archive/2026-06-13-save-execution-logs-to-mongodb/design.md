## Context

The daily Kubernetes vulnerability scanner (`scan.py`) executes standard dependency scans across configured repositories, prints outputs to standard output, and writes final repository findings to MongoDB. Currently, execution logs of the scanning script (e.g., repository initialization steps, execution outputs, and errors) are only outputted to the standard output/error stream, which are ephemeral and lost after log rotation or pod deletion.

This design introduces a mechanism to capture the standard logger outputs during execution and save them as a persistent document in MongoDB at the end of the scanning job.

## Goals / Non-Goals

**Goals:**
- Intercept and buffer all logs generated via the `dependency-scanner` logger.
- Support configuring the target MongoDB collection via the `MONGO_LOGS_COLLECTION` environment variable (default: `execution_logs`).
- Safely persist execution logs to MongoDB on script termination, regardless of whether the run succeeded or failed.
- Ensure the primary scanning operation does not fail if log archiving fails (e.g., database timeout).

**Non-Goals:**
- Streaming logs line-by-line to MongoDB in real time (buffering and saving once at exit is sufficient and reduces db write overhead).
- Redirecting raw standard stdout/stderr of the python process (buffering the logger outputs is cleaner and captures all relevant messages).

## Decisions

### 1. Log Interception Mechanism
We will configure an in-memory buffer (`io.StringIO`) and register a `logging.StreamHandler` writing to this buffer with the same formatting as the console stream. This captures all log outputs generated by `logger.info`, `logger.warning`, and `logger.error` throughout the script lifetime.

*Alternatives Considered:*
- **Redefining `sys.stdout`**: Unnecessarily complex, interferes with lower-level stream management, and might capture binary or unformatted outputs.
- **Buffering Handler**: Standard library `logging.handlers.BufferingHandler` captures raw log records rather than formatted strings, which would require post-formatting during upload. The `StringIO` buffer is simpler and directly captures pre-formatted lines.

### 2. Failure Resiliency & Exit Interception
The saving logic will be wrapped in a `try...except...finally` block inside the script's `main()` function:
```python
def main():
    start_time = datetime.datetime.utcnow().isoformat() + "Z"
    status = "SUCCESS"
    try:
        # scan logic
    except Exception as e:
        logger.error(f"Execution failed: {str(e)}")
        status = "FAILED"
        raise e
    finally:
        save_execution_logs(start_time, status)
```
This guarantees logs are saved even on failure.

### 3. Collection Configuration
We will introduce `MONGO_LOGS_COLLECTION` to `scan.py` and the public `k8s-templates/configmap.yaml` file:
```yaml
MONGO_LOGS_COLLECTION: "execution_logs"
```

## Risks / Trade-offs

- **Risk**: MongoDB is unavailable at the end of the run, preventing log saving and potentially crashing the container.
  - *Mitigation*: Wrap the MongoDB log saving inside a try-except block, logging any insertion error directly to standard output and exiting gracefully. The primary job completion status must not depend on log archiving success.
- **Risk**: Memory growth due to log buffering.
  - *Mitigation*: The daily job scans a limited set of repositories; logs typically remain under 1MB, which has negligible memory impact.
